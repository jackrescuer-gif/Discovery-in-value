## Обзор

Процесс Discovery предназначен для исследования и валидации инициатив, которые прошли PI Planning и имеют определенные бизнес-ценности. Цель процесса — превратить инициативы в работающие в production решения через систематическое исследование, валидацию гипотез, прототипирование, развертывание и эксплуатацию. Финальный этап процесса — эксплуатация решения, где система работает в production и приносит ценность пользователям.

## Входные данные

- **Инициативы после PI Planning** — инициативы, которые прошли планирование Program Increment
- **Определенные бизнес-ценности** — четко сформулированные ожидаемые результаты и метрики успеха
- **Контекст и ограничения** — технические, временные и ресурсные ограничения

## Этапы процесса

### 1. Инициализация Discovery
- Получение инициативы из PI Planning
- Формирование команды Discovery
- Определение scope и временных рамок
- Создание рабочего пространства и инструментов

### 2. Исследование проблемы
- Анализ пользовательских проблем и потребностей
- Изучение существующих решений и конкурентов
- Проведение интервью с пользователями и стейкхолдерами
- Сбор и анализ данных (метрики, аналитика, обратная связь)

### 3. Формулирование гипотез
- Определение ключевых гипотез на основе исследования
- Приоритизация гипотез по важности и риску
- Определение критериев успеха для каждой гипотезы
- Создание карты гипотез (Hypothesis Map)

### 4. Валидация гипотез
- Быстрое тестирование гипотез (эксперименты, опросы, интервью)
- Прототипирование минимальных решений (low-fidelity прототипы)
- Тестирование с пользователями
- Сбор обратной связи и метрик

### 5. Прототипирование решения
- Создание high-fidelity прототипов на основе валидированных гипотез
- Детальная проработка UX/UI
- Техническое прототипирование (proof of concept)
- Интеграционное тестирование
- **Возможный выход:** Готовый продукт или MVP может быть создан на этом этапе

### 6. Валидация решения
- Тестирование прототипа с реальными пользователями
- Измерение ключевых метрик (engagement, conversion, satisfaction)
- Сбор качественной обратной связи
- Итерация на основе результатов

### 6a. Тестирование MVP на части пользователей (опционально)
- **Условие:** Если на этапе Discovery появился готовый продукт или MVP
- Развертывание MVP на ограниченной аудитории (например, 10% пользователей)
- Мониторинг метрик производительности и пользовательского опыта
- Сбор обратной связи от тестовой группы
- Оценка успешности MVP по ключевым метрикам

### 6b. Масштабирование инфраструктуры (если MVP успешен)
- **Условие:** MVP успешно протестирован на части пользователей
- Планирование масштабирования инфраструктуры для 100% пользователей
- Увеличение вычислительных ресурсов и пропускной способности
- Настройка мониторинга и алертинга для production нагрузки
- Подготовка процессов развертывания и отката
- Нагрузочное тестирование инфраструктуры
- **Результат:** Готовность к развертыванию в production

### 7. Подготовка к развертыванию (стандартный путь)
- Документирование решения (спецификации, дизайн-система)
- Оценка технической сложности и рисков
- Планирование внедрения (roadmap, зависимости)
- Передача в команду разработки для реализации

### 8. Развертывание в Production
- Развертывание решения в production-окружении
- Постепенный запуск (25% → 50% → 100% пользователей)
- Мониторинг метрик и производительности
- Обработка инцидентов при необходимости
- Подтверждение стабильной работы системы

### 9. Эксплуатация
- Непрерывный мониторинг системы и метрик
- Сбор обратной связи от пользователей
- Анализ бизнес-метрик и достижения целей
- Планирование итераций и улучшений
- Поддержка и обслуживание системы
- **Результат:** Решение работает в production и приносит ценность

## Выходные данные

### Стандартный путь:
- **Валидированное решение** — прототип, протестированный с пользователями
- **Документация** — спецификации, дизайн, технические требования
- **Метрики валидации** — данные, подтверждающие ценность решения
- **План внедрения** — roadmap и зависимости для production

### Путь через MVP (ускоренный):
- **Готовый MVP продукт** — полностью функциональное решение
- **Результаты тестирования на части пользователей** — метрики успешности (10% аудитории)
- **План масштабирования инфраструктуры** — требования для запуска на 100%
- **Масштабированная инфраструктура** — готовность к production нагрузке
- **Документация по развертыванию** — процессы запуска и мониторинга
- **Развернутое решение в production** — система работает для всех пользователей
- **Эксплуатация** — непрерывная работа и поддержка системы

## Критерии готовности к развертыванию в Production

### Стандартный путь:
✅ Проблема четко определена и валидирована  
✅ Решение протестировано с пользователями и показало положительные результаты  
✅ Бизнес-ценность подтверждена метриками  
✅ Техническая реализуемость оценена  
✅ Документация готова для команды разработки  
✅ План внедрения согласован со стейкхолдерами

### Ускоренный путь через MVP:
✅ MVP продукт готов и функционален  
✅ MVP успешно протестирован на части пользователей (10%) с положительными метриками  
✅ Бизнес-ценность подтверждена метриками тестовой группы  
✅ Инфраструктура масштабирована для 100% нагрузки  
✅ Мониторинг и алертинг настроены для production  
✅ Процессы развертывания и отката готовы  
✅ Нагрузочное тестирование пройдено успешно

## Критерии успешной эксплуатации

✅ Система стабильно работает в production  
✅ Метрики производительности в пределах нормы  
✅ Пользователи активно используют решение  
✅ Бизнес-метрики соответствуют ожиданиям  
✅ Процессы мониторинга и поддержки работают эффективно  
✅ План итераций и улучшений определен  

## Роли и ответственность

- **Product Manager** — координация процесса, приоритизация, коммуникация со стейкхолдерами, принятие решения о масштабировании MVP
- **UX Researcher** — исследование пользователей, валидация гипотез
- **UX/UI Designer** — прототипирование, дизайн решения
- **Tech Lead** — техническая валидация, оценка сложности, планирование масштабирования инфраструктуры
- **DevOps Engineer** — масштабирование инфраструктуры, настройка мониторинга, процессы развертывания
- **Data Analyst** — анализ метрик и данных, оценка успешности MVP на части пользователей
- **Stakeholders** — предоставление контекста, принятие решений о развертывании в production
- **Operations Team** — эксплуатация системы, мониторинг, поддержка (на этапе эксплуатации)

## Метрики успеха процесса

- **Скорость валидации** — время от начала до готовности к production
- **Процент валидированных гипотез** — доля подтвержденных гипотез
- **Качество решения** — метрики пользовательского опыта
- **Эффективность** — соотношение затраченных ресурсов к результату

## Инструменты и методы

- **Исследование**: интервью, опросы, аналитика, конкурентный анализ
- **Прототипирование**: Figma, Sketch, InVision, HTML/CSS прототипы
- **Валидация**: A/B тесты, usability testing, метрики аналитики
- **Документация**: Confluence, Notion, Miro, GitHub Wiki
